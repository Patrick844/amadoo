## 1. Ghost card — image was disappearing in ~100ms (not the full animation)

### What it looked like
You'd swipe right, briefly see the LIKE stamp, and then the card would vanish almost instantly — way before flying off screen.

### Root cause: iOS clips children to `borderRadius` even without `overflow: hidden`

The ghost card was rendered **inside** `deckContainer`, which had `borderRadius: 20`. On iOS, when a parent view has `borderRadius`, it creates a clipping layer that traps any child trying to render outside its bounds. This is true even if the parent does NOT have `overflow: hidden`.

So the ghost card would start at ±60px and try to animate to `SCREEN_W * 1.6` (~600px), but iOS clipped it the moment it crossed the edge of the deck box (~170px). It disappeared after about 100ms of travel, looking like an instant vanish.

```
deckContainer (borderRadius: 20) ← iOS clips here
└── ghost card (absoluteFill + translateX animation)
       → starts at 60px
       → clips at ~170px (edge of deckContainer) ← disappears here
       → target was 600px (never reached)
```

### The fix: render ghost card at the SCREEN level, not inside deckContainer

We moved the ghost card to be a **sibling** of `deckContainer`, not a child of it. It's positioned using `onLayout` — which fires when the deck container lays out and gives us its exact pixel coordinates on screen. We use those to position the ghost card identically, but outside the clipping container.

```
MainView
├── deckContainer (borderRadius: 20) ← only contains actual cards
│   ├── nextNextUser card
│   ├── nextUser card
│   └── SwipeCard (top card, draggable)
│
└── GhostCard ← sibling, NOT inside deckContainer
    position: absolute
    top/left/width/height: copied from deckContainer via onLayout
    translateX: animates freely off screen with no clipping
```

**Code — tracking deck position:**
```tsx
const [deckFrame, setDeckFrame] = useState<{
  top: number; left: number; width: number; height: number
} | null>(null)

<View
  style={styles.deckContainer}
  onLayout={(e) => {
    const { x, y, width, height } = e.nativeEvent.layout
    setDeckFrame({ top: y, left: x, width, height })
  }}
>
```

**Code — ghost card at screen level:**
```tsx
{exitUri && deckFrame && (
  <Animated.View
    style={[
      styles.ghostCard,
      {
        top: deckFrame.top,
        left: deckFrame.left,
        width: deckFrame.width,
        height: deckFrame.height,
      },
      exitCardStyle,
    ]}
    pointerEvents="none"
  >
    <ExpoImage source={{ uri: exitUri }} style={StyleSheet.absoluteFill} ... />
    {exitAction === 'like' && <View style={styles.likeStamp}><Text>LIKE</Text></View>}
    {exitAction === 'dislike' && <View style={styles.nopeStamp}><Text>NOPE</Text></View>}
  </Animated.View>
)}
```

**Ghost card style:**
```js
ghostCard: {
  position: 'absolute',
  borderRadius: 20,
  overflow: 'hidden',
  zIndex: 20,
}
```

The animation is the same: starts at ±60px (always from near center), flies to `direction * SCREEN_W * 1.6` over 600ms with a 50ms initial delay (so React has one frame to render the card before it starts moving).

## 2. Match animation — was completely invisible

### What it looked like
You'd swipe right on Jade, get a match, and nothing would happen — no animation, no overlay.

### Root cause A: `bgOpacity` was wrapping all children

The match overlay used a single `Animated.View` as the root with `opacity: bgOpacity.value`. The background fades from 0→1 over 400ms, and all the child elements (logo, title, photos, buttons) are inside it. In React Native, **a parent's opacity multiplies with its children's opacities**.

So the logo at t=200ms had effective opacity = `bgOpacity (~0.5) × logoOpacity (~0.1) = 0.05`. Completely invisible even though both animations were running fine.

```
Animated.View opacity=bgOpacity (0→1)       ← wraps everything, multiplies
  └── Animated.View opacity=logoOpacity (0→1)  ← effective = 0.5 × 0.1 = 0.05
  └── Animated.View opacity=titleOpacity (0→1)
  └── View (photos)
  └── Animated.View opacity=actionsOpacity (0→1)
```

### Root cause B: overlay was inside the screen View, not a Modal

The match overlay rendered inside the tab screen, meaning the tab bar sat on top of it at full z-index. This could block touch events and even partially obscure the overlay on some layouts.

### The fix: separate background layer + Modal

**Fix 1 — background is its own `absoluteFillObject` layer, NOT the parent:**
```tsx
// Before (wrong — opacity applied to parent, multiplies with children):
<Animated.View style={[styles.matchBg, bgStyle]}>
  <Animated.View style={[logoStyle]}> ... </Animated.View>
  ...
</Animated.View>

// After (correct — background fades independently, children are siblings):
<View style={styles.matchContainer}>
  {/* Background fades in behind everything */}
  <Animated.View style={[StyleSheet.absoluteFillObject, styles.matchBg, bgStyle]} />

  {/* These are NOT children of the fading view — full opacity independent */}
  <Animated.View style={[styles.matchLogoWrap, logoStyle]}> ... </Animated.View>
  <Animated.View style={[styles.matchTitleWrap, titleStyle]}> ... </Animated.View>
  <View style={styles.matchPhotos}> ... </View>
  <Animated.View style={[styles.matchActions, actionsStyle]}> ... </Animated.View>
</View>
```

**Fix 2 — render in a Modal so it sits above the tab bar:**
```tsx
<Modal visible={!!pendingMatch} transparent animationType="none" statusBarTranslucent>
  {pendingMatch && (
    <MatchOverlay
      key={pendingMatch.matchId}   ← key forces fresh mount & fresh useEffect each match
      matchedUser={pendingMatch.user}
      matchId={pendingMatch.matchId}
      myPhoto={user?.photos?.[0]}
      onDismiss={() => setPendingMatch(null)}
    />
  )}
</Modal>
```

**Animation sequence (new timings):**
```
t=0ms     Dark background fades in (350ms)
t=150ms   Amadoo logo drops from above (spring)
t=400ms   "It's a Match!" title rises from below (spring)
t=650ms   Two circular photos fly in from opposite sides (spring)
t=1400ms  "Send a Message" + "Keep Swiping" buttons fade in
```

## 3. Liked section — matched person stayed visible

### What it looked like
After matching with Jade on the swipe tab, you'd open the Liked tab and Jade was still there.

### Root cause: Liked screen only loaded once (on mount)

The liked screen used `useEffect(() => { loadLikes() }, [])` — this only fires when the component first mounts. In Expo Router's tab navigation, the component mounts once and stays mounted. Subsequent visits to the tab don't trigger the effect.

### The fix: `useFocusEffect` to reload on every tab visit

```tsx
// Before:
import { useEffect, useState } from 'react'
useEffect(() => { loadLikes() }, [])

// After:
import { useState, useCallback } from 'react'
import { useFocusEffect } from 'expo-router'

useFocusEffect(
  useCallback(() => {
    loadLikes()
  }, [token])
)
```

Now every time you open the Liked tab it silently refreshes the list. The backend `/likes` endpoint already filters out users you've swiped on — so once you swipe on Jade (matching her), she disappears from the likes list automatically.

## 4. Sharing the app with a friend via Expo Go + EAS Update

### The two problems

1. **Backend URL** — `api.ts` hardcodes your Mac's LAN IP (`192.168.1.108:8000`). A friend on a different network can't reach it.
2. **Frontend bundle** — EAS Update publishes your JS to Expo's CDN so anyone can load it in Expo Go without you running a dev server.

### Why `__DEV__` doesn't help here

In `api.ts`, the original `getBaseUrl()` used:
```ts
if (!__DEV__) return 'https://api.amadoo.com'
return 'http://192.168.1.108:8000'
```

The problem: **in Expo Go, `__DEV__` is always `true`**, even when loading a published EAS Update bundle. So your friend would always hit the LAN IP branch and get connection errors.

### The fix: `EXPO_PUBLIC_API_URL` environment variable

Expo bakes `EXPO_PUBLIC_*` variables into the JS bundle at publish time. We read this first:

```ts
function getBaseUrl(): string {
  if (process.env.EXPO_PUBLIC_API_URL) return process.env.EXPO_PUBLIC_API_URL
  if (!__DEV__) return 'https://api.amadoo.com'
  return 'http://192.168.1.108:8000'
}
```

**For your local dev** (`.env.local` — already created, gitignored):
```
EXPO_PUBLIC_API_URL=http://192.168.1.108:8000
```
Your `npx expo start` sessions continue to use the LAN IP, unchanged.

**For sharing** (passed at publish time):
```bash
EXPO_PUBLIC_API_URL=https://YOUR_NGROK_URL eas update --branch main --message "friend test"
```
The ngrok URL gets baked into the published bundle. Your friend's phone calls your Mac via ngrok.

### How ngrok fits in

ngrok creates a public tunnel to your `localhost:8000`:
```
Friend's phone → internet → ngrok servers → your Mac's port 8000 → FastAPI
```

Your Mac and backend must be running while your friend tests.

## Step-by-step: sharing the app

### One-time setup (do once)

```bash
# 1. Install expo-updates (required for EAS Update)
npx expo install expo-updates

# 2. Log in to your Expo account
npx eas-cli@latest login

# 3. Initialize EAS for this project
#    → creates eas.json
#    → adds projectId to app.json automatically
npx eas-cli@latest init
```

### Every time you want to share

```bash
# Terminal 1 — start backend
cd backend && source .venv/bin/activate
uvicorn main:app --reload --host 0.0.0.0 --port 8000

# Terminal 2 — start ngrok tunnel
ngrok http 8000
# Note the URL it prints, e.g.: https://abc123.ngrok-free.app

# Terminal 3 — publish to Expo (replace with your actual ngrok URL)
cd amadoo
EXPO_PUBLIC_API_URL=https://abc123.ngrok-free.app \
  npx eas update --branch main --message "friend test"
```

EAS prints a preview link. Send it to your friend. They open it in Expo Go.

### What your friend does

1. Install **Expo Go** from the App Store
2. Open the link you sent them — Expo Go loads your app instantly
3. They can sign up and use the real backend through your ngrok tunnel

### Limitations

| Issue | Detail |
|-------|--------|
| Mac must be on | ngrok tunnels to your local machine — if you close your Mac, friends lose backend access |
| ngrok URL changes each session | Free ngrok gives a new URL every time you run it — you need to republish with `eas update` each time |
| Push notifications | Won't work on friends' phones (they'd need to be in your Expo project as testers) |
| One-time setup takes ~10 min | After that, each share is 2 commands |
